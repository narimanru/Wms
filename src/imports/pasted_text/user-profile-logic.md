ТЗ: Личный кабинет пользователя (логика и данные)

1. Общая структура

Страница состоит из 3 независимых функциональных блоков:
	1.	Профиль пользователя
	2.	Смена пароля
	3.	Удаление аккаунта

Каждый блок работает независимо.

⸻

2. Блок «Профиль»

2.1. Данные пользователя

Система должна получать и отображать:

user_id
first_name (Имя)
last_name (Фамилия)
email
email_verified (boolean)
telegram
phone
address
avatar_url (опционально)
created_at
updated_at


⸻

2.2. Логика отображения
	•	Если поле пустое → показывать: “Не указано”
	•	email всегда отображается
	•	если email_verified = true → показывать статус подтверждения

⸻

2.3. Редактирование профиля

При нажатии “Редактировать”:

Открывается форма с полями:

first_name (обязательное)
last_name (необязательное)
telegram (необязательное)
phone (необязательное)
address (необязательное)
avatar (опционально)

E-mail:
	•	не редактируется напрямую
ИЛИ
	•	редактируется через отдельный сценарий (подтверждение) — можно не реализовывать сейчас

⸻

2.4. Валидация
	•	first_name → обязательное
	•	phone → формат (если заполнен)
	•	telegram → строка, без строгой проверки

⸻

2.5. Сохранение

При сохранении:
	•	обновляются данные в БД
	•	обновляется UI без перезагрузки
	•	фиксируется updated_at

⸻

2.6. Ошибки
	•	ошибка API → показать сообщение “Не удалось сохранить данные”
	•	при успехе → “Данные обновлены”

⸻

3. Блок «Смена пароля»

3.1. Входные поля

current_password
new_password
confirm_password


⸻

3.2. Валидация (frontend + backend)

Обязательные:
	•	все поля заполнены

Проверки:
	1.	current_password корректный
	2.	new_password:
	•	минимум 8 символов
	•	минимум 1 буква (a-z)
	•	минимум 1 цифра или спецсимвол
	3.	confirm_password = new_password
	4.	new_password ≠ current_password

⸻

3.3. Логика смены
	1.	Проверка current_password
	2.	Проверка new_password по правилам
	3.	Хеширование нового пароля
	4.	Обновление в БД

⸻

3.4. После успешной смены
	•	вернуть success
	•	опционально:
	•	сбросить все сессии
	•	оставить текущую сессию

⸻

3.5. Ошибки
	•	неверный текущий пароль → “Неверный текущий пароль”
	•	невалидный пароль → “Пароль не соответствует требованиям”
	•	несовпадение → “Пароли не совпадают”
	•	серверная ошибка → “Ошибка при смене пароля”

⸻

4. Блок «Удаление аккаунта»

4.1. Логика активации

Есть чекбокс:

confirm_delete (boolean)

	•	пока false → удаление невозможно
	•	при true → кнопка активна

⸻

4.2. Подтверждение

Перед удалением ОБЯЗАТЕЛЬНО:

2 этапа:
	1.	чекбокс
	2.	подтверждение (modal или повторное действие)

⸻

4.3. Удаление аккаунта

При подтверждении:

Варианты реализации:

Вариант 1 (рекомендуемый):
Soft delete:

deleted_at = timestamp
is_active = false

Вариант 2:
Hard delete:
	•	удаление записи пользователя
	•	удаление связей

⸻

4.4. Что происходит после удаления
	•	пользователь разлогинивается
	•	токены аннулируются
	•	редирект:
→ страница входа / главная

⸻

4.5. Дополнительно

Перед удалением можно:
	•	проверить наличие:
	•	заказов
	•	финансов
	•	остатков
	•	если есть → запретить удаление или предупредить

⸻

5. API (пример)

5.1. Получение профиля

GET /api/user/profile


⸻

5.2. Обновление профиля

PUT /api/user/profile

body:

{
  first_name,
  last_name,
  telegram,
  phone,
  address,
  avatar_url
}


⸻

5.3. Смена пароля

POST /api/user/change-password

body:

{
  current_password,
  new_password
}


⸻

5.4. Удаление аккаунта

DELETE /api/user

или

POST /api/user/delete


⸻

6. Безопасность

Обязательно:
	•	пароли хранить только в hash (bcrypt/argon2)
	•	проверка current_password через hash
	•	rate limit на смену пароля
	•	защита от brute force
	•	удаление токенов при удалении аккаунта

⸻

7. Состояния данных

Профиль:
	•	заполнен
	•	частично заполнен
	•	пустой

Пароль:
	•	валидный
	•	невалидный
	•	ошибка

Аккаунт:
	•	активный
	•	деактивирован
	•	удален

⸻

8. Логи / аудит (желательно)

Фиксировать:
	•	изменение профиля
	•	смена пароля
	•	удаление аккаунта

user_id
action_type
timestamp
ip
device

Система должна уметь:

✔ хранить и обновлять данные пользователя
✔ менять пароль с валидацией
✔ безопасно удалять аккаунт
✔ обрабатывать ошибки
✔ работать без перезагрузки

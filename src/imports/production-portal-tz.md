Ниже — ТЗ для Figma Make на отдельный раздел/кабинет “Производство” (вход для фабрики/цеха), где они видят заказы, принимают в работу, ведут план/факт, меняют статусы, отмечают оплаты и фиксируют отклонения (ткань/брак/недошив). Сделано так, чтобы селлер контролировал, а производство могло делать только своё.

⸻

ТЗ для Figma Make: Кабинет “Производство” (исполнитель)

0) Цель

Дать фабрике/цеху простой интерфейс:
	•	увидеть свои заказы,
	•	принять/отклонить заказ,
	•	вести план vs факт по SKU/размерам,
	•	фиксировать причины отклонений (ткань/брак/усадка),
	•	менять статусы этапов (в работе/ОТК/готово/отгружено),
	•	отмечать оплаты (видеть статус, без доступа к финансам селлера),
	•	общаться/прикладывать документы и фото.

⸻

1) Роли и права (RBAC) — важно для ТЗ

Роль: Production Contractor (Фабрика)

Доступно:
	•	Просмотр только своих заказов и их деталей
	•	Принять/отклонить заказ
	•	Изменять статусы этапов производства
	•	Вносить факт выпуска (по размерам)
	•	Создавать “инцидент/отклонение” (не хватает ткани, брак, недошив)
	•	Загружать документы (акт, накладная, фото партии)
	•	Видеть статус оплат: “ожидается/частично/оплачено” (без сумм или с суммами — решишь)
	•	Комментарии/чат по заказу

Недоступно:
	•	Редактирование карточек товара
	•	Управление КИЗ (только если разрешено отдельно)
	•	Доступ к складам/аналитике WB/другим клиентам
	•	Изменение цены/условий заказа (только через запрос)

⸻

2) Навигация кабинета “Производство”

Sidebar (минимум):
	•	Заказы
	•	Календарь / План
	•	Отгрузки
	•	Документы
	•	Профиль/Настройки

В хедере:
	•	Поиск по заказам
	•	Уведомления (🔔)
	•	Статус “онлайн”

3) Основные экраны (Frames)

Frame P1 — Login (опционально)

Name: Production Portal / Login
	•	Email/Phone
	•	Пароль
	•	“Войти”
	•	“Запросить доступ” (линк)

Верх:

	•	Filters:
	•	Статус: Новые / Приняты / В работе / ОТК / Готово / Отгружено / Закрыто
	•	Срочные (toggle)
	•	Период
	•	KPI mini:
	•	“Новых: 2”
	•	“В работе: 4”
	•	“Просрочено: 1”

Таблица заказов

Колонки:
	•	Заказ (PO-2026-00041)
	•	Клиент/Бренд (Kamilek)
	•	Сезон/Коллекция
	•	Срок готовности (дата + цвет если просрочено)
	•	Объём (шт)
	•	Статус заказа
	•	Статус оплаты (badge: Ожидается / Частично / Оплачено)
	•	Action: Открыть

Строки демо:

Frame P3 — Детали заказа (обзор)

Name: Production Portal / Order Detail / Overview

Header
	•	PO-2026-00041 (badge статуса)
	•	Клиент: Kamilek
	•	Срок готовности: 25 марта
	•	Объём: 1050 шт
	•	Buttons (в зависимости от статуса):
	•	Принять заказ (primary)
	•	Отклонить / Запросить изменения
	•	Сообщить о проблеме (⚠️)
	•	Загрузить документ

Карточки-статусы (4)
	•	Статус заказа: Новый / Принят / В работе …
	•	Статус этапа: Крой / Пошив / ОТК / Упаковка
	•	План/Факт: 1050 / 0 (с прогрессбаром)

Блок “Ключевые условия” (read-only)
	•	Кто печатает КИЗ: “Мы/Селлер/Фулфилмент” (только просмотр)
	•	Упаковка: пакет/бирка/стикер (чек-лист)
	•	Комментарии клиента (если есть)

⸻

Frame P4 — Состав заказа (План vs Факт по размерам)

Name: Production Portal / Order Detail / Items (Plan vs Fact)

Таблица по SKU/размерам:

Колонки:
	•	SKU / Наименование
	•	Размер
	•	План (шт)
	•	Факт (шт) — editable
	•	Отклонение (±)
	•	Причина (dropdown) — если отклонение ≠ 0
	•	Комментарий
	•	Статус строки (в работе/готово)

Пример причин (dropdown):
	•	Недостаток ткани
	•	Брак/дефекты
	•	Усадка/перекос
	•	Нет фурнитуры
	•	Замена материала (по согласованию)
	•	Другое

UX-логика:
	•	Если факт меньше плана → поле “Причина” становится обязательным
	•	Если факт больше плана → требует подтверждения “перевыпуск”
	•	Кнопка: Сохранить факт
	•	Кнопка: Предложить замену (например перераспределить по размерам)

Важно: добавить подсказку:
“В одежде факт может отличаться от плана — фиксируйте причину, чтобы клиент видел прозрачность.”

⸻

Frame P5 — Этапы производства (workflow)

Name: Production Portal / Order Detail / Workflow

Визуал: timeline/kanban внутри заказа.

Этапы (карточки):
	1.	Принято
	2.	Крой
	3.	Пошив
	4.	ВТО
	5.	ОТК
	6.	Упаковка
	7.	Готово к отгрузке
	8.	Отгружено

Для каждого этапа:
	•	статус (Not started / In progress / Done)
	•	дата начала/окончания
	•	ответственный (dropdown внутри фабрики)
	•	комментарий + вложения (фото)

Кнопка: Сменить этап → Следующий

⸻

Frame P6 — Оплата (только статусы)

Name: Production Portal / Order Detail / Payments

Показываем прозрачно, но безопасно.

Вариант A (без сумм):
	•	Предоплата: “Ожидается / Получена”
	•	Остаток: “Не выставлен / Ожидается / Получен”
	•	Даты: когда оплачено

Вариант B (с суммами, если разрешишь):
	•	Сумма предоплаты, сумма остатка, итого

Кнопки для производства:
	•	Подтвердить получение (если ты хочешь, чтобы они отмечали факт поступления)
	•	Запросить оплату (создаёт уведомление клиенту)

⸻

Frame P7 — Инциденты / проблемы (очень важно)

Name: Production Portal / Order Detail / Issues

Список инцидентов по заказу:
	•	“Не хватает ткани на 120 шт размера M” — статус: Ожидает решения
	•	“Брак партии 18 шт” — статус: Решено

Кнопка: Создать инцидент

Модалка “Создать инцидент”

Поля:
	•	Тип: ткань / брак / сроки / фурнитура / другое
	•	SKU/размер (select)
	•	Кол-во затронуто (шт)
	•	Предложение решения (chips):
	•	Сдвинуть срок
	•	Снизить объём (факт меньше плана)
	•	Перераспределить по размерам
	•	Запросить доп. ткань/фурнитуру
	•	Комментарий
	•	Вложения (фото)

CTA: Отправить клиенту

⸻

Frame P8 — Отгрузка (готовность и документы)

Name: Production Portal / Shipments

Список отгрузок по заказам:

	•	“Загрузить акт/фото коробов”

⸻

Frame P9 — Календарь / План (производственная загрузка)

Name: Production Portal / Calendar

Вид: неделя/месяц
	•	Заказы как блоки по срокам
	•	Цвет по статусу/срочности
	•	Быстрый фильтр “Просрочено”

Цель: фабрика видит загрузку и сроки.

⸻

4) Важные модалки/состояния

Модалка “Принять заказ”
	•	“Подтвердите, что сроки и объём реалистичны”
	•	чек:
	•	“Материалы доступны”
	•	“Сроки подтверждаю”
	•	кнопка: Принять

Модалка “Запросить изменения”
	•	выбрать: сроки / объём / материалы
	•	причина
	•	предложение (например “-30 шт из-за ткани”)
	•	отправить клиенту

⸻

5) План/факт: ключевая логика (обязательно отразить в UI)
	•	В каждой строке размера отображать:
	•	План
	•	Факт
	•	Отклонение
	•	Если отклонение есть → “Причина” обязательна
	•	На уровне заказа: агрегированный прогресс
	•	“Факт 980 / План 1050 (−70)”
	•	“Причина: недостаток ткани (подтверждение в Issues)”

⸻

6) Уведомления (что показать в демо)

В кабинете производства уведомления:
	•	“Новый заказ PO-2026-00041 — требуется принятие”
	•	“Клиент ответил по инциденту: согласовано уменьшение на 30 шт”
	•	“Предоплата получена — можно запускать”

⸻

7) Демо-данные (встроить в прототип)

Заказ: PO-2026-00041, Kamilek, 1050 шт, срок 25 марта
В Items:
	•	Pants Rib: S 300, M 400, L 300
	•	Top Basic: S 200, M 180, L 150

Сценарий “реальной жизни”:
	•	факт по Pants Rib M = 380 (−20), причина: “недостаток ткани”
	•	факт по Top Basic L = 140 (−10), причина: “брак/дефекты”
Создать инцидент на 30 шт.

⸻

8) Прототип-сценарий (1–2 минуты)
	1.	Orders List → открыть PO-2026-00041
	2.	Overview → Принять заказ
	3.	Items → внести факт (380 вместо 400) → выбрать причину → сохранить
	4.	Issues → создать инцидент “не хватает ткани 20 шт” → отправить
	5.	Workflow → поставить этап “Пошив: In progress”
	6.	Payments → показать “Предоплата получена” (badge)
	7.	Shipments → “Готово к отгрузке” (чек-лист)

⸻

9) Что “можно” и “нельзя” в доступе (подписи для дизайна)

На экранах, где селлерское:
	•	скрыть кнопки “Редактировать цену/условия”
	•	скрыть поля “КИЗ банк” (если не разрешено)
	•	оставить только “просмотр требований по маркировке/упаковке”
